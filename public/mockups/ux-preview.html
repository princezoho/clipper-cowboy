Save indicators & the yellow coverage strip

Two changes to make the cataloger feel honest about what it's doing. (A) Persistent + transient feedback for every save and one-shot action. (B) A translucent yellow band on the editor timeline and pool cards showing what's already been clipped from a source video, so you can see coverage at a glance.

Status Code already exists for the timeline strips (currently emerald) — this preview shows the proposed yellow visual and refinements. We'll swap colors and ship the rest based on your notes.

C

Editing existing clips & saving in-progress work

You're clipping a 30-minute source over several days. This section answers: where does progress live, how do you re-edit clips you've already exported, and what happens if you close the tab mid-edit. Three layers: drafts (unsaved local state), clips (committed sidecar files), and a session activity log.

C.0 — Save ≠ re-export — and we know which files are already there Test it ↗ passing

Saving metadata never triggers ffmpeg. The system tracks which video files exist on disk and surfaces missing ones as an opt-in repair action — not an automatic re-render.

To trigger the missing-file banner manually: mv "$PROJECT_DIR/clips/<any-clip>.mp4" <name>.mp4.bak then reload the Library tab. Rename back to clear. Drop a file like orphan-test.mp4 into the clips folder to trigger the orphan banner.

Detection: on every /api/library fetch, the server stat()s each meta.path and flags missing ones; reverse-scans the clips folder for files with no sidecar. The Re-render missing button is the only way to trigger ffmpeg from this state. Adopt creates a sidecar from the orphan file's filename + duration.
Contract: metadata saves write JSON only. Video files are only ever produced by an explicit Export or Re-export. Closing the app, reopening tomorrow, or switching sources never re-renders anything.

Feedback prompts

1. Yellow at the proposed opacities (25% / 55%) — readable enough on dark frames, too dim, or too loud? visual

If too loud at the top edge, drop the top marker to 40%. If too dim on bright frames, bump the fill to 30%.

2. Should delete toasts (character, scene, clip) always include Undo, or only for destructive bulk actions? toast

Undo extends auto-dismiss to 8s. Inline-only undo means less visual noise but harder to recover from an "oh wait" moment.

3. Should autosave fire on blur in addition to 800ms debounce, or only on debounce? autosave

Blur+debounce = more requests but feels safer. Debounce-only = quieter network but you can lose state if you Cmd-W within the debounce window.

4. Overlapping clips — offset bands (B.2) or stacked tooltip on a single merged stripe? timeline

Offset is more legible at a glance, merged is cleaner on dense source files. Could also be a setting.

5. Should clicking a yellow strip on the pool card jump straight into Re-export mode, or just open the editor at that timestamp? flow

Editor at timestamp is less destructive; Re-export mode is faster if that's almost always what you'd do anyway.

6. Persistent header status — show only when dirty/saving, or always (even on green Saved)? header

Always-on builds trust ("yes, the app is tracking your edits"). Conditional reduces chrome but introduces flicker.

7. Anything missing? (e.g. "Saved" indicator on collection cards, autosave for character names, etc.) open

Add notes here and I'll fold them in before the implementation pass.

8. Drafts (C.2) — auto-restore silently on reopen, or always show the amber "Draft restored" banner? draft

Silent restore is friction-free but invisible if you forgot you had a draft. Banner is louder but explicit. Could also default to silent + show the banner only if the draft is >1 day old.

9. Coverage % (C.3) — surface as a number ("57%"), a donut, both, or only at extremes (e.g. only show "Fully covered")? coverage

Both is informative but adds a row of chrome to every Pool card. Donut-only is glanceable, number-only is sortable in the UI later. The "fully covered" badge is the most decision-relevant signal.

10. Activity log (C.4) — keep it as a small popover, promote to a full sidebar, or skip the in-app surface and just write the .jsonl file? activity

Popover is cheap and useful for "wait what did I just do." Sidebar is overkill until there's filtering / scrubbing. .jsonl-only ships fastest if you'd never actually open the panel.

11. Library integrity (C.0) — should the "X clips have missing files" banner appear automatically at the top of Library, or be tucked behind a small icon in the header? integrity

Auto-banner is loud but explicit — you never miss a missing file. Tucked-icon (e.g. a tiny ⚠ in the header that opens the same actions) is quieter but easier to ignore. Could also default to auto-banner only when count > 0 on first load of a session.
